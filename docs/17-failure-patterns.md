# Common Failure Patterns

| Pattern | Symptom | Fix |
|---|---|---|
| Kitchen-sink session | Unrelated tasks mixed, context polluted | `/clear` between tasks |
| Correction loop | Same mistake corrected 2+ times | `/clear` + rewrite the prompt with what you learned |
| Over-specified CLAUDE.md | Claude ignores some rules | Prune ruthlessly — if Claude does it right without the rule, delete it |
| Trust-then-verify gap | Plausible implementation with hidden edge cases | Always give Claude a runnable check |
| Infinite exploration | Claude reads hundreds of files, context full | Scope investigations narrowly or use subagents |
| No stopping condition | Loop runs forever or halts randomly | Define a verifiable success criterion before starting |
| Uncapped headless run | Expensive, hard to debug after the fact | Set `--max-turns` and `--max-budget-usd` |
| Reviewer bias | Claude reviews code it wrote, misses issues | Always use a subagent for final review |
| Polling loop | Cron fires every N minutes just to check if work exists, burning tokens on empty runs | Use event-driven triggers (webhook, file watcher, CI event) so the loop only fires when there is actually something to do |
| Loop as wrong unit | Designing individual loops in isolation when the real system needs multiple loops coordinated at a higher level | Identify the system of loops first (what monitors what, who hands off to whom) before designing each loop's internals |
| Cost runaway | A loop without a hard spend cap runs for days and accumulates thousands of dollars in API costs before anyone notices ($47k in 11 days is a real example) | Always set `--max-budget-usd` before scheduling a headless loop; treat missing budget cap as a deployment blocker |
| Provider lock-in | Hard-coding a single model provider into the loop means one outage or policy change triggers both an emergency and a compliance review simultaneously | Design agent loops with model-agnostic interfaces from the start; use provider abstraction (OpenAI-compatible endpoints) so you can swap the model without rewriting the loop |
| Cognitive surrender | Accepting AI output without retaining your own judgment — delegating not just execution but evaluation. Engineers who copy-paste AI output score under 40% on comprehension tests vs 65%+ for those using AI assistively (Osmani, May 2026) | Treat every AI output as a first draft; apply engineering judgment before merging; distinguish offloading (delegate with retained oversight) from surrender (accept unchecked) |
| Orchestration tax | Human attention becomes the bottleneck in multi-agent loops — managing feedback from parallel agents consumes more cognitive load than the original task would have | Apply backpressure: limit parallel agents, batch their outputs, review asynchronously; the orchestration tax grows faster than the parallelism benefit |
| Intent debt | Undocumented design decisions become the most expensive technical debt in agentic workflows — agents cannot fabricate authentic rationale and silently fill gaps with plausible-sounding explanations | Document every design decision before launching the loop (use `CLAUDE.md` or a `DECISIONS.md`); treat undocumented intent as a defect, not a style choice |
| Dark factory | Fully autonomous loop with zero human checkpoints — correct output is assumed, errors compound silently, and there is no escalation path when things go wrong | Define at least one mandatory human-in-the-loop gate per major phase; never deploy a loop with zero checkpoints (see [Human-in-the-Loop](14-human-in-the-loop.md)) |
| Missing circuit breaker | Loop continues executing despite repeated failures or stalled progress — burns budget and compounds errors without auto-halting | Add a circuit breaker: track consecutive failures and exit with an error if N failures accumulate or no measurable progress is made in M turns; treat circuit breaker absence as a deployment blocker |
| Reward hacking | Agent satisfies the validation criteria without solving the underlying problem — it learns to game the evaluator rather than produce correct output | Use objective, external evidence as the verifier (test results, curl timings, diff counts) rather than AI-generated self-assessments; rotate or harden the evaluation criteria so the agent cannot pattern-match to past passes |
| Context pollution | A long conversation accumulates every bad decision, reversal, and dead end — the agent is no longer reasoning from a clean slate; it is carrying "accumulated wreckage" and hallucination becomes a predictable outcome, not a random error | Reset context deliberately between loop iterations: encode learnings externally (skills, rules, PROGRESS.md) before clearing, then start the next iteration fresh. Throwing away context feels like losing progress — it actually protects the signal. (Martin Dilger, Jun 2026) |
| Context drift | The original goal erodes across many iterations as the agent adapts to intermediate states — by iteration N the loop is optimising for something subtly different from what was specified at iteration 0 | Re-read the TRIGGER and SCOPE from the Loop Contract at the start of every iteration; treat any deviation from the original goal as a stopping condition, not a natural evolution |
| Amplification effect | Each loop iteration adds small defensive responses to local failures, gradually making the system less understandable while appearing more robust — a slow accumulation of complexity that obscures logic without improving reliability | Periodically review loop-generated code at the system level, not just per-iteration output; treat unexplained defensive code as a signal that the verifier is too narrow (Armin Ronacher, Jun 2026) |
| Cognitive dependency | Becoming reliant on AI not just as a tool but as integral to understanding and maintaining the codebase — loop-generated code becomes unmaintainable without permanent AI participation | Require that every loop-produced feature is understood and owned by a human engineer; treat AI-only-legible code as a defect, not a benefit (Armin Ronacher, Jun 2026) |
| State rot | The loop's STATE.md references merged PRs, closed tickets, and completed tasks that no longer exist — the loop acts on ghosts, wasting tokens and sometimes taking harmful actions on stale references | Prune resolved items at the start of every run; validate STATE.md IDs against the live API before acting on them; treat any state reference older than 24h as unverified ([cobusgreyling/loop-engineering](https://github.com/cobusgreyling/loop-engineering), Jun 2026) |
| Verifier theater | The evaluator "approves" output without actually running the check — same model context as the implementer, no test command executed, verdict based on reasoning rather than evidence | The verifier's default stance must be REJECT; it must execute test commands and include the command output in its verdict; a verifier that approves without evidence is indistinguishable from no verifier ([cobusgreyling/loop-engineering](https://github.com/cobusgreyling/loop-engineering), Jun 2026) |
| Notification fatigue | The loop pings a channel on every run rather than only on actionable findings — team learns to ignore all loop notifications, including genuine failures | Notify only on delta: new failures, newly resolved items, or state changes that require human action; a run with nothing new should produce no output ([cobusgreyling/loop-engineering](https://github.com/cobusgreyling/loop-engineering), Jun 2026) |
| Infinite fix loop | A loop that fails a check retries indefinitely — never escalating, never stopping — burning budget and compounding corruption in the codebase with each bad attempt | Set a hard attempt cap per failure (e.g. 3 attempts for CI Sweeper) and escalate to a human inbox after the cap is reached; treat absence of an attempt cap as a deployment blocker ([cobusgreyling/loop-engineering](https://github.com/cobusgreyling/loop-engineering), Jun 2026) |
